## 3. レイヤ構成

### 3.1 Source Content

- `content/`
  - 原稿本文、画像、添付ファイルの置き場
  - 入力記法は MediaWiki風Wiki記法、標準Markdown、Obsidian記法の混在を許容する
- `content/samples/`
  - parser 検証や表示確認に使うサンプル原稿
  - 本番公開対象からは除外する
  - `content/{任意名フォルダ}/`
- 本番公開時に使う
- サブフォルダの存在を許容し、contentフォルダ内はすべて再帰的走査の対象となる

### 3.2 Import / Normalize

- 今後追加する取り込み層
  - `content/` 配下の原稿とアセットを収集する
  - MediaWiki風Wiki記法、標準Markdown、Obsidian記法の差異を吸収する
  - frontmatter とテンプレート由来の情報を共通メタデータへ正規化する
  - 本文は見出し、段落、リンク、テンプレート呼び出しなどの小さな共通表現へ落とし込む
- テンプレート変換
  - MediaWiki風テンプレートと表示の間の変換規則は、この層の別モジュールとして扱う
  - テンプレートごとの差し替えや追加は可能にする
  - ただし下流へ渡す公開用データ形式は必要最小限の共通モデルに保つ

### 3.3 Data

- `src/data/articles.js`
  - トップページで表示する記事データの仮置き
  - 将来は `content/` 解析結果や JSON 出力に差し替える
- `src/data/site-config.js`
  - カテゴリ定義、参加導線、実装メモなど固定的なUI設定

### 3.4 Model

- `src/lib/home-page-model.js`
  - 並び替え
  - 選り抜き記事選定
  - 検索
  - カテゴリ件数集計
  - 画面表示用モデル組み立て

UIに依存しない処理をここへ寄せ、Node の組み込みテストから直接検証する。

### 3.5 View

- `src/lib/renderers.js`
  - HTMLエスケープ
  - 記事カード描画
  - 検索結果描画
  - カテゴリカード描画

描画専用に閉じることで、DOM操作とテンプレート生成の責務を分離する。

### 3.6 Bootstrap

- `src/main.js`
  - 初期描画
  - 検索イベント
  - 記事選択イベント
  - ランダム選択イベント

## 8. 記事インポートと公開フロー

このプロジェクトで目指す記事インポートは、原稿と公開物を分離する。

- 原稿
  - 参加者が各自のローカル環境で編集する Wikiテキスト / Markdown 原稿
  - GitHub 上ではレビュー対象の正本として扱う
- 公開物
  - 管理者レビューとマージを通過した原稿から自動生成される HTML 群
  - 生成成果物であり、手編集の対象にしない

### 8.1 目標フロー

1. 参加者が Obsidian や任意のテキストエディタで原稿を編集する
2. 原稿を GitHub へ Pull Request として送る
3. 管理者または編集者が内容、リンク、frontmatter、公開可否をレビューする
4. 承認された原稿を `main` へマージする
5. CI または静的サイト生成処理が Markdown 原稿を公開ページへ変換する
6. 生成済みサイトを GitHub Pages へデプロイする
7. 読者が公開ページを閲覧できる状態になる

### 8.2 予定している入力と出力

- 入力
  - `content/` 配下の原稿ファイル
  - 添付画像や関連アセット
  - `content/samples/` 配下のサンプル原稿
- 中間処理
  - frontmatter 解析
  - テンプレート由来メタデータの正規化
  - MediaWiki風Wiki記法 / Markdown / Obsidian 記法の正規化
  - 内部リンク解決
  - alias と正規ページの対応付け
  - リダイレクト候補と曖昧さ回避候補の抽出
  - バックリンク用の参照グラフ生成
  - 未解決リンクの抽出
  - Git 履歴からの更新情報抽出
  - 目次、検索用データ、一覧ページ用メタ情報の抽出
- 出力
  - 記事ごとの HTML
  - 一覧、検索、カテゴリ向けの補助データ
  - ページごとのバックリンク情報
  - リダイレクトページまたは正規ページ案内用データ
  - 曖昧さ回避ページ用データ
  - 未作成記事リンク一覧と未作成記事ページ用データ
  - ページごとの更新履歴メタデータ
  - GitHub Pages へ配信する静的ファイル一式

- 現在のトップページと記事詳細表示は `src/data/articles.js` の仮データを描画している
- 記事詳細ルーティング、内部リンク解決、バックリンク表示、未作成記事案内は仮データ上で実装済み

## 9. バックリンクと未作成記事の設計

### 9.1 バックリンク

- ビルド時に全ページの内部リンクを収集し、リンク元からリンク先への有向グラフを作る
- 各ページの公開データには、そのページを参照しているリンク元一覧を付与する
- 公開画面では本文末尾またはサイド領域にバックリンク一覧を表示する
- バックリンクは本文リンクとは別の補助ナビゲーションとして扱う
- 特定見出し名の配下へ自動挿入するようなマジック解釈は MVP では採らず、表示位置はレイアウト側で固定する

### 9.2 未作成記事

- 内部リンクの解決時にリンク先が存在しない場合、そのリンクを未解決リンクとして保持する
- 未解決リンクはビルド失敗の理由にはせず、赤リンク用の状態として出力へ反映する
- 赤リンク遷移先として、対象記事名ごとの未作成記事ページを生成するか、同等の案内ルートを持つ
- 未作成記事ページには、参照元ページ一覧と近似候補ページを表示できるようにする

### 9.3 alias、リダイレクト、曖昧さ回避

- `aliases` はリンク解決候補として扱うが、一意解決できる場合と衝突する場合を分ける
- 一意解決できる alias は正規ページへのリダイレクトまたは同等の案内へ変換する
- 複数ページにまたがる名称は曖昧さ回避ページを生成し、「XXはこちら」の案内を機械生成できるようにする
- これらの案内は本文への手書きテンプレートより、frontmatter と生成処理に寄せる

### 9.4 必要な生成物

- `backlinks.json` のようなページ単位参照マップ
- `missing-pages.json` のような未解決リンク一覧
- `redirects.json` のような alias と正規ページの対応表
- `disambiguation.json` のような曖昧さ回避候補一覧
- `history.json` のような更新履歴メタデータ
- 必要に応じて未作成記事ページ用の HTML またはルーティング情報

### 9.5 実装時の注意

- ファイル名変更や alias によりリンク先が変わるため、リンク解決はタイトルだけでなく alias も考慮する
- 同名ページや類似名ページの衝突ルールを先に定義する
- 未解決リンクをすべてエラーにすると草案運用と相性が悪いため、警告と可視化を基本とする
- Git 履歴抽出を使う場合、CI では shallow clone を避けるか、必要な履歴だけ取得する

## 10. Markdown / Wiki互換と管理向け機能

### 10.1 記法互換の方針

- 執筆記法は MediaWiki風Wiki記法を基礎とし、標準MarkdownとObsidian互換記法を入力互換として受ける
- 新機能は本文中の特定文字列や見出し名へのマジック依存ではなく、frontmatter とビルド時メタデータで表現する
- 入力記法の差異は取り込み時の正規化層で吸収し、下流では共通の小さなデータモデルだけを扱う
- これにより、既存のMediaWiki風原稿も、Obsidian や通常の Markdown ツールで書かれた原稿も、最終的に同じ公開フローへ乗せられる

### 10.2 テンプレートの扱い

- テンプレートやインフォボックスは必要性を認めるが、MVP では独自Wiki記法の全面導入は避ける
- MediaWiki風テンプレートは入力互換として受けるが、そのまま下流へ流すのではなく、テンプレート変換ハンドラで共通データモデルへ落とし込む
- まずは frontmatter から生成できる定型表示を優先し、必要なテンプレートだけを限定的にハンドラ実装する
- 曖昧さ回避や「XXはこちら」の案内も、本文の手書きテンプレートではなく生成処理側で差し込む

### 10.3 管理者向け保守情報

- 孤立ページ、行き止まりページ、未解決リンク、リンク密度などは、主に管理者向けの保守レポートとして扱う
- これらは公開サイト本体より、管理アプリ、ビルドレポート、または Obsidian / グラフ表示側へ寄せる方針を取れる
- 読者向けUIと管理者向け分析UIは分離して考える

### 10.4 ページ履歴

- 読者向けには最終更新日と簡易更新履歴を出し、詳細な差分は GitHub 側へ委譲できる
- 生成元は Git の履歴とし、必要に応じて PR 情報やコミットメッセージを要約へ利用する
