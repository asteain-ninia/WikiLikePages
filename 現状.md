> 本分析に基づき、設計文書 (spec.md v0.2→v0.3, implementation-architecture.md v0.3) を更新済み。
> Phase 1 の一部 (テスト追加・インライン書式・テンプレートプラグイン方式) を実装済み。

## 1. アプリの全体像

GitHub Pages で公開する Wikipedia 風の静的 Wiki 基盤。
編集は Obsidian 等の外部エディタに委譲し、PR マージを契機に自動デプロイする構成。

### 現在の動作形態

- **SPA (Single Page Application)** として動作。ハッシュルーティング (`#!article/...`, `#!missing/...`, `#!disambiguation/...`) で画面遷移する
- 記事データは `content/` の Markdown 原稿を `npm run build:content` で事前に JS モジュール (`src/data/generated/content-data.js`) へ変換し、ブラウザで読み込む
- 記事ごとの個別 HTML は生成しない。全記事データを 1 つの JS に載せてクライアントで描画している

## 4. 設計上の課題

### 4.1 SPA 構成の限界

現在の構成は全記事データを 1 つの JS ファイルに格納し、クライアントで描画する SPA。

- **スケーラビリティ**: 記事数が増えると `content-data.js` が肥大化し、初回ロードが重くなる
- **SEO**: ハッシュルーティングはクローラーに不利。OGP メタタグも記事ごとに設定できない
- **直リンク**: `#!article/...` 形式の URL は共有時に不安定になりやすい
- 仕様では「原稿から生成された HTML を正とする」(spec 8.8) と明記されており、最終的には記事ごとの HTML 生成に移行する必要がある

### 4.2 ~~テンプレート処理の空白~~ → 解決済み

~~テンプレートプラグイン方式を実装済み。`基礎情報 国` (インフォボックス) と `架空` (ノーティス) のハンドラが動作する。~~

### 4.3 ~~記事本文のインライン書式~~ → 部分解決

インライン書式 (太字・斜体・コード・MediaWiki書式) はセクション本文で保持され、描画層で HTML に変換されるようになった。ただしブロックレベル書式 (リスト・テーブル・外部リンク) の HTML 変換は未実装。

### 4.4 build:site が単なるファイルコピー

`build-site.mjs` は `index.html` と `src/` を `dist/` にコピーするだけ。バンドル、ミニファイ、HTML 生成は一切行わない。GitHub Pages に載るのは開発時と同じファイル構成。
